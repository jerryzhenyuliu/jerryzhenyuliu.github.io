Six sorting algorithms, and the O(n²) gap that hit harder than I expected

I just finished my first “proper” CS project: implement six classical
sorting algorithms and actually measure how they stack up. I knew going
in that “O(n log n) beats O(n²)”. The textbook says so. But I wanted to
see the gap with my own eyes, not just know it as a fact.

Seeing the numbers on my own laptop turned out to be a different kind of
understanding.

The lineup

Six algorithms, same function signature so the experiment could treat
them uniformly:

Similar single-file implementations for insertion sort, merge sort,
quicksort (with random pivot and with 3-way Dutch-flag partition), and
counting sort. Every one of them returns (sorted_list, comparison_count) —
so I could count comparisons separately from wall time.

Six input distributions

The textbook complexity is for “random” input, but I wanted to probe the
edges:

  1. Uniform random — the default
  2. Already sorted
  3. Reverse sorted
  4. Almost sorted (5% of elements swapped)
  5. Many duplicates (values only in [0..9])
  6. Block-sorted (sorted chunks concatenated in shuffled order)

Data generation uses a fixed seed so runs are reproducible.

What the numbers look like

I ran each (algorithm × distribution × n) seven times and kept the
median. Here’s the big-picture log-log plot on uniform random input:

The lines you want to look at:

  • Red and yellow (bubble, insertion) — steeper. Each decade of n adds
    two decades of time. That’s slope ≈ 2, which is O(n²) on log-log.
  • Green and blue (merge, quick) — shallower. Each decade of n adds
    slightly more than one decade of time. Slope ≈ 1, consistent with
    O(n log n).
  • Purple (counting sort) — almost flat at first, then a sudden
    hop. More on that below.

The gap in raw numbers

Time at n = 10,000 on uniform random input:

algorithm time (ms) comparisons
bubble 10,840 49,991,840
insertion 4,344 25,059,528
merge 33 120,370
quick (random) 45 150,539
counting 28 0

Bubble sort took almost eleven seconds on a list of ten thousand
integers. Merge sort took 33 milliseconds. That’s a factor of ~330.
“O(n²) is slower” stops being an abstract phrase when one version is
literally waiting for lunch to finish and the other one is done before
you blink.

The comparison counts tell the same story without the noise of machine
timing. Merge sort does ~120K comparisons on that input; bubble sort
does 50 million.

The interesting edge: input shape matters a lot

Here’s the same six algorithms across all six input distributions, at
n = 1000:

A few things jumped out:

Bubble sort has one superpower

On already-sorted input, bubble sort does one pass. A single sweep of
O(n) comparisons, the early-termination flag never trips, it exits.
On that one specific input, the thing everyone calls “the worst sort”
beats everything except counting sort. Useful reminder that “best” is
never a property of an algorithm alone — it’s always algorithm times
input.

Quicksort’s weakness is duplicates, not size

The “many duplicates” column is where I caught something real. Normal
random-pivot quicksort uses roughly as many comparisons on duplicate-heavy
data as on random data — each duplicate still has to decide “less than
or greater than pivot” and lands in one of the two sides. But 3-way
quicksort (Dutch flag partition) drops the duplicates into an “equal”
middle bucket in one pass, and the recursion skips over them entirely.
On all-equal input, that’s O(n) instead of O(n log n).

This is the reason most modern sort implementations use some form of
3-way partition — they get linear time on highly skewed real-world data
almost for free.

Counting sort wins only when it wins

Counting sort is O(n + k) where k is the value range. At n = 100,000
and k = 100, it’s three times faster than merge sort. At k = 10,000
it’s slower. At k = 1,000,000 it runs out of memory trying to allocate
a million-element bucket array. So it’s not a “better” algorithm — it’s
a different tool that’s useful under specific conditions (small,
known integer range).

The part I thought I understood but didn’t

I’d read “O(n log n) beats O(n²) for large n” a dozen times before
writing any of this. But reading it and watching bubble sort take
11 seconds on your own machine while merge sort finishes in a blink are
completely different experiences.

There’s also a subtler thing: the constants matter. At n = 10 or n = 100,
bubble sort can actually beat merge sort because merge sort’s recursion
overhead dominates for tiny inputs. Real-world sort implementations
(CPython’s Timsort, Java’s dual-pivot quicksort) hybridize — they
switch to insertion sort for small subarrays. That only makes sense
once you’ve measured the crossover point yourself.

What’s next
